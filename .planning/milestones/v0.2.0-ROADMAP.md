# Milestone v0.2.0: milestone

**Status:** ✅ SHIPPED 2026-04-23
**Phases:** 1-6
**Total Plans:** 21

## Overview

DMJedi v0.2.0 shipped the first full milestone beyond the initial compiler core. It added production-oriented SQL dialect support, real integration testing against DuckDB and SQLGlot, a single-document LSP server, MCP and JSON automation surfaces, and streaming output plus release assets for Spark Declarative generation.

## Phases

### Phase 1: Generator Infrastructure

**Goal**: Establish factory-pattern generator registry, dialect-aware type mapping, and template refactoring for multi-target SQL generation.
**Depends on**: none
**Plans**: 4 plans

Plans:

- [x] 01-01: test scaffolds for factory-pattern registry and dialect-aware type mapping
- [x] 01-02: registry factory pattern, generator constructor updates, and system type entries
- [x] 01-03: DDL template refactor to route system column types through `map_type()`
- [x] 01-04: Ruff lint gap closure on Phase 1 files

**Details:**
Phase 1 established the extensibility base for the milestone: generators are now instantiated through a parameterized registry, SQL system types flow through centralized dialect-aware mapping, and the SQL Jinja templates no longer hardcode system column types.

### Phase 2: SQL Dialects

**Goal**: Deliver DuckDB, Databricks SQL, and PostgreSQL dialect support with hash keys and correct identifier quoting, building on the type map and registry infrastructure from Phase 1.
**Depends on**: Phase 1
**Plans**: 3 plans

Plans:

- [x] 02-01: hash expression builder, quoting, CHAR(64) hash types, and dialect-aware DDL output
- [x] 02-02: staging view templates and generator wiring for all supported physical entity types
- [x] 02-03: dialect snapshot tests, hash tests, and all-entity fixture coverage

**Details:**
Phase 2 shipped SQL dialect support for DuckDB, Databricks SQL, and PostgreSQL, including dialect-aware hash expressions, ANSI-safe identifier quoting, staging views, and committed snapshot coverage across the full DV2.1 entity surface.

### Phase 3: Integration Testing

**Goal**: Validate generated SQL against real databases and enforce quality gates — DuckDB execution tests, canonical fixtures, SQLGlot syntax validation, and coverage thresholds.
**Depends on**: Phase 2
**Plans**: 4 plans

Plans:

- [x] 03-01: uv/bootstrap setup and 85% coverage gate
- [x] 03-02: canonical fixtures and SQL execution helpers
- [x] 03-03: DuckDB behavioral assertions and Databricks SQLGlot parsing gate
- [x] 03-04: gap-closure work to finish full generated DuckDB execution coverage

**Details:**
Phase 3 turned the dialect work into a real quality gate: generated SQL is exercised against in-memory DuckDB, Databricks SQL is parsed by SQLGlot, and the repository now enforces a minimum coverage threshold.

### Phase 4: LSP Server

**Goal**: Implement a Language Server Protocol server providing real-time diagnostics, completions, hover info, go-to-definition, and document symbols for DVML files in any LSP-compatible editor.
**Depends on**: Phase 1
**Plans**: 4 plans

Plans:

- [x] 04-01: current-document diagnostics foundation, `pygls` server bootstrap, and `dmjedi lsp` CLI wiring
- [x] 04-02: conservative keyword and same-document entity-reference completions
- [x] 04-03: same-document hover, go-to-definition, document symbols, and final regression gate
- [x] 04-04: semantic diagnostic and completion-trust gap closure from UAT

**Details:**
Phase 4 shipped a single-document LSP implementation with parse and lint diagnostics, conservative completions, hover, go-to-definition, and document symbols. A later gap-closure plan added semantic parent validation so invalid same-document relationships no longer appear semantically valid in-editor.

### Phase 5: MCP Server & Structured Output

**Goal**: Expose DMJedi capabilities (validate, generate, explain) as MCP tools for AI workflows, and add `--format json` for structured CLI output consumable by LLMs and CI/CD.

**Details:**
Phase 5 added machine-readable automation surfaces. The CLI now emits stable JSON envelopes, and DMJedi exposes `validate`, `generate`, and `explain` as MCP tools over stdio through FastMCP.

### Phase 6: Streaming & Release

**Goal**: Add streaming pipeline support to the Spark Declarative generator and prepare the v0.2.0 release with changelog, examples, and tagged GitHub release.

## Milestone Summary

**Decimal Phases:**

- None

**Key Decisions:**

- Use a factory-pattern generator registry so dialect and mode become generator parameters instead of target-specific branching.
- Keep the v0.2.0 LSP implementation single-document-only to ship a reliable first cut before workspace-aware behavior.
- Expose automation contracts through shared application services so CLI JSON and MCP remain aligned.
- Keep Spark streaming as a mode on the existing `spark-declarative` target rather than creating a parallel target.
- Treat GitHub release publication as a documented manual edge rather than hidden automation.

**Issues Resolved:**

- SQL dialect support is no longer blocked by hardcoded system column types.
- Generated SQL is now validated through both real execution and parser-level integration gates.
- The former LSP stub was replaced with a functioning stdio server and its semantic-completion trust gap was closed.
- DMJedi now exposes structured output for both CLI automation and MCP-based AI workflows.

**Issues Deferred:**

- Future milestone work still includes Star Schema support, loading patterns, IDE packaging, and possible broader LSP/editor integration.
- GitHub release publication remains manual by design.

**Technical Debt Incurred:**

- `.planning/REQUIREMENTS.md`, `.planning/ROADMAP.md`, and `.planning/STATE.md` drifted during the milestone and were corrected at archive time.
- Phases 2, 3, 5, and 6 still have non-compliant Nyquist validation files even though their implementation and verification are complete.
