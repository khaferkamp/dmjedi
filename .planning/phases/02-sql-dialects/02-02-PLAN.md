Create 7 staging view templates and wire them into the SqlJinjaGenerator so that every DDL entity (hub, satellite, link, nhsat, nhlink, effsat, samlink) automatically produces a companion staging view computing hash keys via dialect-appropriate SHA-256 expressions.

Create 7 staging view Jinja2 templates. Each staging view is a CREATE OR REPLACE VIEW that selects from a source table, computes hash keys from business keys using the `hash_expr` callable (injected as Jinja2 global in Plan 01), and provides columns matching the DDL table.

IMPORTANT: The `hash_expr` Jinja2 global is a Python callable that takes a `list[str]` of column names. In Jinja2 templates, build the list using Jinja2's `map` filter: `hub.business_keys | map(attribute='name') | list`. This avoids Python list comprehension syntax which Jinja2 does not support (Research assumption A1).

All templates must use the `| q` filter for identifier quoting and include the Databricks ANSI mode header.

**staging_hub.sql.j2:**
```jinja2
-- Staging view: {{ hub.name }}
-- Generated by DMJEDI
{% if dialect == 'databricks' -%}
-- NOTE: Requires ANSI_MODE=true and spark.sql.doubleQuotedIdentifiers=true
{% endif -%}

CREATE OR REPLACE VIEW {{ ('stg_' ~ hub.name) | q }} AS
SELECT
    {{ hash_expr(hub.business_keys | map(attribute='name') | list) }} AS {{ (hub.name ~ '_hk') | q }},
    CURRENT_TIMESTAMP AS {{ "load_ts" | q }},
    'dmjedi' AS {{ "record_source" | q }}{% if hub.business_keys %},{% endif %}
{%- for bk in hub.business_keys %}
    {{ bk.name | q }}{% if not loop.last %},{% endif %}
{%- endfor %}
FROM {{ ('src_' ~ hub.name) | q }};
```

**staging_satellite.sql.j2:**

Satellites need two hashes: (1) the parent hub's hash key (from the parent_ref business keys -- but at the satellite level we don't have direct access to the parent hub's business keys). The staging view for a satellite should hash the satellite's own columns for hash_diff, and reference the parent's hash key column as a passthrough from the source. The pattern is: the source staging area for a satellite has the parent_hk already computed (by the hub's staging view), plus the satellite's descriptive columns. The satellite staging view computes `hash_diff` from its own columns.

```jinja2
-- Staging view: {{ sat.name }} (parent: {{ sat.parent_ref }})
-- Generated by DMJEDI
{% if dialect == 'databricks' -%}
-- NOTE: Requires ANSI_MODE=true and spark.sql.doubleQuotedIdentifiers=true
{% endif -%}

CREATE OR REPLACE VIEW {{ ('stg_' ~ sat.name) | q }} AS
SELECT
    {{ (sat.parent_ref ~ '_hk') | q }},
    CURRENT_TIMESTAMP AS {{ "load_ts" | q }},
    CURRENT_TIMESTAMP AS {{ "load_end_ts" | q }},
    'dmjedi' AS {{ "record_source" | q }},
    {{ hash_expr(sat.columns | map(attribute='name') | list) }} AS {{ "hash_diff" | q }}{% if sat.columns %},{% endif %}
{%- for col in sat.columns %}
    {{ col.name | q }}{% if not loop.last %},{% endif %}
{%- endfor %}
FROM {{ ('src_' ~ sat.name) | q }};
```

**staging_link.sql.j2:**

Links hash their own business key as a composite of all hub reference hash keys. The staging view passes through hub ref hash key columns and computes the link's own hash key from them.

```jinja2
-- Staging view: {{ link.name }}
-- Generated by DMJEDI
{% if dialect == 'databricks' -%}
-- NOTE: Requires ANSI_MODE=true and spark.sql.doubleQuotedIdentifiers=true
{% endif -%}
{%- set ref_hk_names = [] -%}
{%- for ref in link.hub_references -%}
  {%- set _ = ref_hk_names.append(ref ~ '_hk') -%}
{%- endfor %}

CREATE OR REPLACE VIEW {{ ('stg_' ~ link.name) | q }} AS
SELECT
    {{ hash_expr(ref_hk_names) }} AS {{ (link.name ~ '_hk') | q }},
    CURRENT_TIMESTAMP AS {{ "load_ts" | q }},
    'dmjedi' AS {{ "record_source" | q }},
{%- for ref in link.hub_references %}
    {{ (ref ~ '_hk') | q }}{% if not loop.last or link.columns %},{% endif %}
{%- endfor %}
{%- for col in link.columns %}
    {{ col.name | q }}{% if not loop.last %},{% endif %}
{%- endfor %}
FROM {{ ('src_' ~ link.name) | q }};
```

IMPORTANT: For link hash keys, the hash is computed from the hub reference hash key *column names* (e.g., `Customer_hk`, `Product_hk`), not from raw business keys. The Jinja2 expression `link.hub_references | map('~ "_hk"') | list` may not work as intended because Jinja2's `map` filter with a string argument calls that string as a filter on each item. Instead, use a Jinja2 workaround. The simplest approach: build the list in the template using a namespace or just use the string concatenation approach:

Actually, the cleanest approach for link staging is: pass the hub_references through, and the hash is computed from the hub ref hk column names. Use this pattern instead:

```jinja2
{%- set ref_hk_names = [] -%}
{%- for ref in link.hub_references -%}
  {%- set _ = ref_hk_names.append(ref ~ '_hk') -%}
{%- endfor -%}
    {{ hash_expr(ref_hk_names) }} AS {{ (link.name ~ '_hk') | q }},
```
